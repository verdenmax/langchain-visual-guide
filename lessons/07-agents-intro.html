Agent 入门

把前三课（消息 + 模型 + 工具）组装并自动化，就是 Agent。 一个函数 create_agent 帮你编排"思考 → 调工具 → 看结果 → 再思考"的循环，直到任务完成。

🕵️ 生活类比
Agent 就像一个会用工具的助理：你交代任务，他自己判断"需要先查个天气"， 拿起工具电话拨号，看到结果后再决定"够了，可以回答了"还是"还得再查一个"。 你不用一步步指挥——循环由他自己跑。

一行创建 Agent

from langchain.agents import create_agent

agent = create_agent(
    model="openai:gpt-5.1",        # 模型（字符串或模型对象都行）
    tools=[get_weather],            # 一组工具
    system_prompt="你是天气助手",  # 可选的系统设定
)

result = agent.invoke(
    {"messages": [{"role": "user", "content": "北京和上海哪个更热？"}]}
)
print(result["messages"][-1].content)   # 最终答案
🔬 细节 / 代码对应
  • create_agent 在 langchain_v1/langchain/agents/factory.py， 由 agents/__init__.py 导出。
  • 它的输入/输出都是一个带 "messages" 键的状态字典——即 AgentState。
  • 核心签名（节选）： 
    def create_agent(
    model: str | BaseChatModel,
    tools=None,
    *,
    system_prompt=None,
    middleware=(),         # 可插拔的中间件（第 13 课）
    response_format=None,  # 结构化输出
    ...
)

Agent 循环：它到底在转什么？

这张图是 Agent 的灵魂。注意它就是第 6 课那个工具往返，只是自动重复， 直到模型不再请求工具为止：

① 模型思考
看当前消息，决定下一步
→
要调工具？→ ② 执行工具
运行后把 ToolMessage 加回消息，回到 ①
不调工具？→ ③ 结束
返回最终 AIMessage
🌍 宏观理解
  • Agent = 模型 + 工具 + 一个自动循环。循环的"判断条件"很简单： 模型回复里还有没有 tool_calls？有就执行并继续，没有就结束。
  • 你只管声明"用哪个模型、给哪些工具、什么系统设定"，循环的编排交给框架。
  • 底层其实是一张 LangGraph 状态图（节点："model" 与 "tools"）——这是第 13 课的内容，现在只需理解它在转一个循环。

🔍 深入理解

展开下面的卡片，深入 create_agent 的三个常用能力。

1 system_prompt：怎么给 Agent 定"人设"和规则 点击展开详解
🧪 示例
agent = create_agent(
    model, tools=[get_weather],
    system_prompt="你是严谨的天气助手。只用工具返回的数据，不要编造。",
)
❓ 为什么必要
它会作为对话最前面的 SystemMessage 注入，约束 Agent 的语气、边界、调用工具的策略。 没有它，模型只能凭默认行为发挥，容易跑偏或乱编。
✅ 优点
一个字符串就能统一规范整个 Agent 的行为，无需在每条用户消息里重复叮嘱。
2 response_format：让 Agent 最终输出结构化数据 点击展开详解
🧪 示例
from pydantic import BaseModel
class Answer(BaseModel):
    city: str
    temp_c: float

agent = create_agent(model, tools=[get_weather], response_format=Answer)
# Agent 跑完循环后，给你一个 Answer 对象，而不只是文本
❓ 与 with_structured_output 的区别
第 5 课的 with_structured_output 约束单次模型调用的输出； 这里的 response_format 约束整个 Agent 循环跑完后的最终结果——适合"先用工具查、最后给结构化答案"的场景。
✅ 优点
Agent 既能多步用工具，又能把最终结论交成强类型对象，直接对接下游程序。
3 middleware：在循环里插钩子（预告第 13 课） 点击展开详解
🧪 用途举例
create_agent(..., middleware=[...]) 可以在"调用模型前后""执行工具前后"插入逻辑： 改写请求、人审(human-in-the-loop)、日志、限流、敏感词过滤等。
❓ 为什么必要
真实产品需要在 Agent 的标准循环里加"非功能性"逻辑。若没有扩展点，就只能去改核心循环代码——既危险又难维护。
✅ 优点 & 🔀 其他方案
优点：行为可插拔，核心循环保持稳定（第 13 课看它如何映射到 LangGraph 节点）。 其他方案：用回调（callbacks，第 14 课）做观测，但 middleware 能真正改变流程，回调主要用于观察。

🎛️ 进阶专题：如何"驯服"一个 Agent

自动循环很强，但生产环境需要给它装上刹车与护栏：限制循环次数、关键操作要人审、 出错能重试或降级。这些大多通过中间件（middleware）声明式装上，全部来自 langchain.agents.middleware。

需求中间件 / 机制
防止无限循环 / 失控ModelCallLimitMiddleware · ToolCallLimitMiddleware · recursion_limit
关键操作要人批准HumanInTheLoopMiddleware
出错自动重试ModelRetryMiddleware · ToolRetryMiddleware
主模型挂了降级备用ModelFallbackMiddleware
工具太多先筛一遍LLMToolSelectorMiddleware

🔍 Agent 控制 · 深入理解

展开下面的卡片，逐个吃透关键护栏。

1 防止失控：循环与调用次数上限 点击展开详解
🧪 两道防线
from langchain.agents.middleware import ModelCallLimitMiddleware

agent = create_agent(model, tools, middleware=[
    ModelCallLimitMiddleware(max_calls=8),   # 业务级：最多调 8 次模型
])
# 硬保护：图的递归上限（兜底，超过直接报错）
agent.invoke(input, config={"recursion_limit": 25})
❓ 为什么必要
模型可能陷入"反复调同一个工具"的死循环，既烧钱又卡死。 recursion_limit 是 LangGraph 的兜底硬上限；ModelCallLimit / ToolCallLimit 则是更贴业务的软上限，可控制"最多思考几轮、某工具最多调几次"。
✅ 优点
两层防线：软上限给出友好行为（如到达上限就收尾回答），硬上限保证绝不无限跑。
2 人在回路（HITL）：危险操作前先批准 点击展开详解
🧪 示例
from langchain.agents.middleware import HumanInTheLoopMiddleware

agent = create_agent(model, tools=[delete_database], middleware=[
    HumanInTheLoopMiddleware(...),   # 调用敏感工具前暂停，等人确认
], checkpointer=saver)               # HITL 需要可暂停/恢复，故要 checkpointer
❓ 为什么必要
有些工具是不可逆的高风险操作（删数据、转账、发邮件）。让 Agent 在执行前暂停等人点头， 是生产环境的安全底线。它依赖图的"中断/恢复"能力（第 13 课）。
✅ 优点
把"自动"与"可控"结合：日常自动跑，关键节点交回人类决策，且暂停期间状态被持久化、可随时恢复。
3 健壮性：重试与模型降级 点击展开详解
🧪 示例
from langchain.agents.middleware import (
    ToolRetryMiddleware, ModelFallbackMiddleware,
)
agent = create_agent(primary_model, tools, middleware=[
    ToolRetryMiddleware(max_retries=2),        # 工具抖动自动重试
    ModelFallbackMiddleware(backup_model),     # 主模型失败→切备用
])
❓ 为什么必要
网络抖动、限流 429、厂商宕机都会偶发。Retry 吸收瞬时错误， Fallback 在主模型彻底不可用时切到备用，保证服务连续。
✅ 优点 & 🔀 其他方案
优点：健壮性以"装中间件"的方式声明式获得，核心逻辑不变。 其他方案：在单次模型调用层用 model.with_retry()（第 5 课），但 Agent 级中间件覆盖整个循环更全面。
💡 设计亮点
  • Agent = 模型 + 工具 + 循环，判据简单到只有一句话：还有没有 tool_calls。
  • 你只声明"用什么"，循环的编排交给框架——这就是声明式的威力。
✅ 本课要点
  • 输入/输出都是 {"messages": [...]} 状态字典（AgentState）。
  • 循环判据：AIMessage 里还有没有 tool_calls。这把第 4–6 课串成了闭环。

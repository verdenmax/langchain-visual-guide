工具 Tools

工具（Tool）让模型突破"只会聊天"的边界——查数据库、调 API、做计算。 神奇之处在于：你只写一个普通 Python 函数，LangChain 自动把它变成模型能理解、能"请求调用"的工具。

🧰 生活类比
模型像个聪明但被关在房间里的顾问：它知识渊博，却拿不到实时信息、碰不到外部系统。 工具就是你递进房间的一部部专用电话——一部打给天气台、一部打给数据库。 你贴上"这部电话能干嘛、要拨什么号"的说明（schema），顾问就会在需要时请你帮他拨号。

把函数变成工具：@tool

最常用的方式是 @tool 装饰器。它会自动从你的函数里提取三样东西：

from langchain_core.tools import tool

@tool
def get_weather(city: str) -> str:
    """查询某城市的当前天气。"""
    return f"{city} 晴，25°C"
函数名
get_weather
→ 工具名
+
docstring
"查询天气"
→ 工具描述
+
类型注解
city: str
→ 参数 schema
🔬 细节 / 代码对应
  • @tool 定义在 core/tools/convert.py，把函数包装成一个 BaseTool。
  • 实际生成的是 StructuredTool（core/tools/structured.py）， 它用 create_schema_from_function 从函数签名自动推断出 pydantic 参数 schema。
  • 所以"写好类型注解 + docstring"很重要——它们直接决定模型看到的工具说明。

工具自己也能被调用

包装后的工具是个对象，有 name、description、 args_schema，并且可以像函数一样 invoke：

print(get_weather.name)         # "get_weather"
print(get_weather.description)  # "查询某城市的当前天气。"
print(get_weather.invoke({"city": "北京"}))  # "北京 晴，25°C"

模型如何"使用"工具：完整往返

关键认知：模型不会自己执行工具，它只会"请求"。真正的执行由你的程序（或 Agent）完成。 这一来一回是整个工具机制的核心：

1

绑定

model.bind_tools([get_weather])：把工具的 schema 告诉模型。

2

请求

模型回一条 AIMessage，其中 tool_calls=[{name, args, id}]——"我想调 get_weather('北京')"。

3

执行

你的程序按请求真正运行 get_weather.invoke(args)，拿到结果。

4

回传

把结果包成 ToolMessage（带上对应的 tool_call_id）追加到消息列表。

5

再问

把更新后的消息再喂给模型，它据此给出最终自然语言回答。

🌍 宏观理解
这套"请求 → 执行 → 回传 → 再问"的循环，手写也能实现，但很繁琐。 第 7 课的 create_agent 就是把这整个循环自动化。 也就是说：工具 + 这个循环 = Agent。

手写一次完整往返（可跑）

from langchain.chat_models import init_chat_model
from langchain_core.messages import HumanMessage, ToolMessage

model = init_chat_model("gpt-5.1").bind_tools([get_weather])

msgs = [HumanMessage("北京天气怎么样？")]
ai = model.invoke(msgs)                    # ① 模型请求调用工具
msgs.append(ai)

for call in ai.tool_calls:                  # ② 你来执行（可能不止一个）
    result = get_weather.invoke(call["args"])
    msgs.append(ToolMessage(result, tool_call_id=call["id"]))  # ③ 回传，靠 id 配对

final = model.invoke(msgs)                 # ④ 再问 → 自然语言答复
print(final.content)

这正是 create_agent 内部循环的手写版——把它自动化（再加多轮与错误处理）就是一个 Agent。

🔍 深入理解

展开下面的卡片，深入三个进阶用法。

1 @tool 的几种写法与定制 点击展开详解
🧪 示例
# 1) 最简：函数名=工具名，docstring=描述
@tool
def get_weather(city: str) -> str:
    """查询天气"""
    ...

# 2) 自定义名字 + 直接返回（不再回模型）
@tool("weather", return_direct=True)
def get_weather(city: str) -> str: ...

# 3) 用 pydantic 显式定义参数 schema
@tool(args_schema=WeatherInput)
def get_weather(city: str) -> str: ...
❓ return_direct 是什么
默认工具结果会回传给模型再让它总结。设 return_direct=True 后， 工具一旦执行，其结果直接作为最终输出返回给用户，不再多走一轮模型——省时省 token，适合"查到即答"的工具。
✅ 优点
同一个装饰器覆盖从"零配置"到"精细控制"的全谱：能自动推断时省心，需要时又能完全接管 name/description/schema。
2 ToolMessage 不只是文本：content / artifact / status 点击展开详解
🧪 字段一览
ToolMessage(
    content="北京 晴 25°C",        # 给模型看的文本结果
    tool_call_id="call_abc",       # 与请求配对的 id
    artifact={"raw": {...}},       # 不给模型、留给程序用的原始数据
    status="success",            # success / error
)
❓ artifact 有什么用
有时工具返回的"完整原始数据"（如一张图、一个大 JSON）不适合塞给模型（费 token 又没必要）， 但你的程序后续要用。artifact 就是"给程序、不给模型"的那一份， 靠 response_format="content_and_artifact" 启用。
✅ 优点
把"给模型的摘要"和"给程序的原始数据"分离，既省上下文又不丢信息；status 还能让 Agent 感知工具失败。
3 写好工具的关键：类型注解 + docstring 点击展开详解
🧪 好 vs 差
# ✅ 好：类型清晰、描述明确，模型才知道何时/如何调用
@tool
def search_orders(user_id: str, status: str) -> list:
    """按用户 id 和订单状态查询订单。status 可选: paid/shipped。"""

# ❌ 差：无注解、无说明，模型只能瞎猜
@tool
def f(a, b): ...
❓ 为什么必要
模型看不到函数体，它只能根据"工具名 + 描述 + 参数 schema"决定要不要调、传什么参。 注解和 docstring 直接变成模型看到的"说明书"，写得含糊，模型就用得糟。
✅ 经验法则
每个参数都加类型注解；docstring 一句话说清"做什么 + 关键参数取值范围"； 工具名用动词短语（get_weather 而非 gw）。

🧰 进阶专题：让工具更强大、更安全

真实工具不只是"输入→输出"。它们常需要：出错时优雅处理、读取 Agent 的运行时状态、 危险操作前人审、工具太多时先筛选。这些大多在 Agent / 工具节点层提供。

需求机制
工具抛异常不崩溃ToolNode 错误处理 · ToolRetryMiddleware · status="error"
工具读取 Agent 状态InjectedState · InjectedToolCallId
危险工具先人审HumanInTheLoopMiddleware
工具太多先筛一遍LLMToolSelectorMiddleware

🔍 工具进阶 · 深入理解

展开下面的卡片，逐个吃透进阶能力。

1 工具出错了，Agent 怎么不崩 点击展开详解
🧪 错误被转成消息，而非异常
在 Agent 里，工具抛异常时 ToolNode 会把它捕获并转成一条 ToolMessage （status="error"），让模型"看到错误"并自行调整： 
ToolMessage(content="Error: 城市不存在", status="error", tool_call_id=...)
# 模型读到后可改参数重试，或换个工具，而不是整个程序崩掉
❓ 为什么这样设计
Agent 是自动循环，单个工具失败不该让整条流程挂掉。把错误反馈给模型， 它往往能自我纠正（如换个查询）。要"自动重试"则叠加 ToolRetryMiddleware。
✅ 优点
把"异常"变成"对话信号"，让 Agent 具备一定的自愈能力；是否捕获/如何捕获可在 ToolNode 配置。
2 让工具读取 Agent 状态：InjectedState 点击展开详解
🧪 示例
from typing import Annotated
from langchain.tools import InjectedState

@tool
def summarize_history(state: Annotated[dict, InjectedState]) -> str:
    """总结当前对话。"""
    msgs = state["messages"]   # 工具能拿到 Agent 状态
    ...
❓ 为什么必要 / 妙在哪
有些工具需要"对话上下文"或"运行时数据"才能工作，但这些不该由模型来填参数 （模型也填不对）。InjectedState 标记的参数会被框架自动注入， 且不会出现在给模型看的 schema 里——模型根本不知道它存在。 类似地 InjectedToolCallId 注入当前调用 id。
✅ 代码对应
InjectedState / InjectedToolCallId 从 langchain.tools 导出 （底层来自 langgraph.prebuilt）。
3 危险工具人审 + 工具太多先筛 点击展开详解
🧪 两个中间件
from langchain.agents.middleware import (
    HumanInTheLoopMiddleware,     # 执行敏感工具前暂停等批准
    LLMToolSelectorMiddleware,    # 工具很多时，先让模型挑出相关子集
)
agent = create_agent(model, tools=large_toolset, middleware=[
    LLMToolSelectorMiddleware(...),
    HumanInTheLoopMiddleware(...),
])
❓ 各自解决什么
工具太多（几十上百个）会撑爆上下文、降低选择准确率， LLMToolSelectorMiddleware 先按当前任务筛出相关工具再交给模型。 危险工具则用 HumanInTheLoopMiddleware 加人审闸门。
✅ 优点
安全性与可扩展性都以"装中间件"获得：工具集再大也能用，危险操作也可控，核心代码不变。
4 工具集 Toolkit 与注入参数基类 InjectedToolArg 点击展开详解
🧪 BaseToolkit：把一组相关工具打包
数据库、文件系统等常有"一组配套工具"。BaseToolkit 把它们打成一包： 
toolkit = SomeDatabaseToolkit(db=db)
agent = create_agent(model, tools=toolkit.get_tools())   # 一次性给一组工具
位置：core/tools/base.py。
❓ InjectedToolArg 是什么
它是所有"注入参数"的基类（tools/base.py）：第 6 课的 InjectedState、InjectedToolCallId（:1404）都继承它。 被它标记的参数由框架注入、不出现在给模型的 schema 里。
✅ 为什么有用
Toolkit 让"成套能力"即插即用；InjectedToolArg 提供统一机制，把"运行时数据"安全喂给工具而不暴露给模型。
💡 设计亮点
  • 函数签名即 schema：写好类型注解 + docstring，模型的"说明书"自动生成。
  • 模型只请求、不执行：执行权留在你手里——这是安全与可控的关键边界。
✅ 本课要点
  • @tool 把普通函数变成工具，自动提取名字 / 描述(docstring) / 参数 schema(类型注解)。
  • 底层是 StructuredTool，用 pydantic 从函数签名推断 schema。
  • 模型只请求调用（写在 AIMessage.tool_calls），执行靠你的程序，结果用 ToolMessage 回传。
  • 进阶：错误转消息（自愈）、InjectedState（读状态不暴露给模型）、人审、工具筛选。
  • 把这个往返自动化，就得到了 Agent —— 下一课登场。

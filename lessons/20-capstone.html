端到端实战：拼一个客服 Agent

前面 19 课，你学的都是一个个零件。这一课把它们拼成一台完整的机器： 一个能查知识库、查订单、按用户身份个性化、带护栏、给结构化答复的客服 Agent。 你会看到——每一块，都是前面学过的东西。

🧩 生活类比
之前是在认识螺丝、齿轮、电机；现在我们把它们组装成一台机器。 神奇的是：组装过程几乎没有"新知识"，全是把学过的零件插到对的接口上—— 这正是 LangChain 统一抽象的最终回报。

零件 → 课程对照地图

这个 Agent 用到的每一块，都能追溯到前面的某一课：

零件作用来自
init_chat_model聊天模型第 5 课
@tool（查订单）调用外部系统第 6 课
create_retriever_toolRAG 知识库工具第 17 课
system_prompt人设与规则第 16 课
AgentMiddleware审计 / 护栏第 18 课
context_schema注入 user_id第 19 课
response_format结构化答复第 5 课
create_agent把以上接成循环第 7 / 13 课

完整代码：拼起来其实就这么短

from dataclasses import dataclass
from pydantic import BaseModel
from langchain.chat_models import init_chat_model
from langchain.agents import create_agent
from langchain.agents.middleware import AgentMiddleware
from langchain_core.tools import tool, create_retriever_tool
from langchain.tools import ToolRuntime

# ① 运行时上下文（第 19 课）：每次请求带 user_id，不进消息
@dataclass
class Ctx:
    user_id: str

# ② 工具（第 6 课）：查订单，用 runtime 读当前用户
@tool
def get_order_status(order_id: str, runtime: ToolRuntime) -> str:
    """查询订单状态。"""
    return orders_db.status(runtime.context.user_id, order_id)

# ③ RAG（第 17 课）：把知识库检索器变成一个工具
kb_tool = create_retriever_tool(retriever, "search_kb", "检索退换货等政策")

# ④ 自定义中间件（第 18 课）：每轮审计 + 护栏
class AuditMiddleware(AgentMiddleware):
    def before_model(self, state, runtime):
        log.info("user=%s turns=%d", runtime.context.user_id, len(state["messages"]))
        return None

# ⑤ 结构化答复（第 5 课）
class Reply(BaseModel):
    answer: str
    resolved: bool

# ⑥ 组装（第 7、13 课）：一个 create_agent 把以上全部接起来
agent = create_agent(
    model=init_chat_model("openai:gpt-5.1"),                # 第 5 课
    tools=[get_order_status, kb_tool],                       # 第 6、17 课
    system_prompt="你是友好的客服，先查资料再回答，不要编造。",   # 第 16 课
    middleware=[AuditMiddleware()],                          # 第 18 课
    context_schema=Ctx,                                      # 第 19 课
    response_format=Reply,                                   # 第 5 课
)

# ⑦ 跑起来：context 带身份，输出是结构化的 Reply
result = agent.invoke(
    {"messages": [{"role": "user", "content": "我的订单 A123 到哪了？"}]},
    context={"user_id": "u_42"},
)
print(result["structured_response"])   # Reply(answer=..., resolved=...)

一次真实请求，数据怎么流

1

进入：带身份的请求

用户消息 + context={"user_id": "u_42"} 进入 Agent 图。

2

中间件先跑

AuditMiddleware.before_model 记录这一轮（读到 user_id，但不污染对话）。

3

模型决定用哪个工具

模型看到"订单"→ 请求 get_order_status("A123")；若问政策则请求 search_kb。

4

工具执行（带上下文）

get_order_status 通过 runtime.context.user_id 只查"这个用户"的订单，结果回灌为 ToolMessage。

5

循环 + 结构化收尾

模型据工具结果作答；因设了 response_format，最终给出 Reply(answer, resolved)。

💡 设计亮点
  • 几乎没有"新代码"，全是拼装：每个零件都是前面学过的 Runnable / 抽象，插到 create_agent 的对应参数上即可。
  • 关注点分离得很干净：身份走 context、护栏走 middleware、知识走 RAG 工具、格式走 response_format——彼此不纠缠，可单独替换/测试。
  • 可平滑长大：要记忆就加 checkpointer，要人审就加 HumanInTheLoopMiddleware，要多 Agent 就把它当子图——都不用重写核心。

🔍 深入理解

展开下面的卡片，看这个 Agent 如何继续长大与被测试。

1 加记忆：让它记住多轮对话 点击展开详解
🧪 一行加持久化
from langgraph.checkpoint.memory import InMemorySaver
agent = create_agent(..., checkpointer=InMemorySaver())
agent.invoke(inp, config={"configurable": {"thread_id": "u_42"}})  # 同 thread 记住上下文
（第 13 课）按 thread_id 区分会话，自动存取历史。
✅ 配合上下文管理
历史变长时，叠加 SummarizationMiddleware（第 4 课）自动摘要，控制 token。
2 给 UI 流式：显示"正在查订单…" 点击展开详解
🧪 用 stream_mode
for ev in agent.stream(inp, context=ctx, stream_mode="updates"):
    ...  # 每个节点的增量：模型/工具步骤，驱动进度提示
要逐字打字机用 "messages"；要细到每一步事件用 astream_events（第 14 课）。
3 测试：不联网也能验证整个 Agent 点击展开详解
🧪 用假模型替换真模型
from langchain_core.language_models.fake_chat_models import GenericFakeChatModel
test_agent = create_agent(model=GenericFakeChatModel(messages=iter([...])), tools=[...])
# 断言它在该调工具时调了工具、最终结构化字段正确
（第 15 课）单测禁网、用假模型，快速稳定地验证 Agent 的编排逻辑。
✅ 分层测
工具单独测（纯函数）、链/Agent 用假模型测编排、真模型留给集成测试——三层各司其职。
✅ 本课要点 & 全书结业
  • 一个真实 Agent = 把学过的零件插到 create_agent 的对应参数上：model / tools / system_prompt / middleware / context_schema / response_format。
  • 身份走 context、护栏走 middleware、知识走 RAG 工具、格式走 response_format——关注点干净分离。
  • 要长大很平滑：checkpointer 加记忆、stream_mode 加 UI、假模型做测试。

🧪 自测 · 想一想为什么这么设计

1. 客服 Agent 里：“身份走 context、护栏走 middleware、知识走 RAG 工具、格式走 response_format”。这种“各走各的机制”体现了什么？
  1. 机制太多
  2. 关注点干净分离：每类需求用最合适、互不纠缠的机制实现，改一处不影响其它
  3. 炫技
  4. 凑参数
看答案与解析 点击展开
答案：B。把不同维度的需求（身份/安全/知识/格式）分派到正交机制上，是整套教程设计哲学的集大成——也是系统能长大而不乱的原因。
2. 课里说这个 Agent“要长大很平滑”：加 checkpointer 就有记忆、加 stream_mode 就有 UI、换假模型就能测试。这得益于什么？
  1. 运气好
  2. 各能力都是正交、可插拔的零件，新增能力是“插上一个参数/组件”，而非重写
  3. 代码少
  4. 模型强
看答案与解析 点击展开
答案：B。当系统由“可插拔的正交零件”组成，演进就是“加零件”而非“动手术”。可组合性最终兑现为可演进性。
3. create_agent 把 model/tools/system_prompt/middleware/context_schema/response_format 都做成“参数”。这种“用参数声明一切”图什么？
  1. 参数多显得强
  2. 声明式装配：你把学过的零件“插到对应参数上”就拼出 Agent，意图清晰、组合自由
  3. 让调用更长
  4. 没有意图
看答案与解析 点击展开
答案：B。把各能力暴露成正交参数，使用者用“声明”而非“编码”来组装系统——这是声明式设计在 API 层的体现。
4. 走完主线你会发现：实战不是学新东西，而是“把学过的零件插到对应位置”。这说明这套框架的设计成功在哪？
  1. 功能多
  2. 概念一致、可组合：前面每课学的零件，到实战时无缝拼合，没有“实战要另学一套”的断层
  3. 文档好
  4. 模型强
看答案与解析 点击展开
答案：B。当“局部概念”能直接组合成“完整系统”而无须额外胶水，说明抽象选对了。可组合性让学习曲线和工程曲线对齐。
💭 发散思考（没有标准答案，动手或动脑想想）
  • “身份/护栏/知识/格式各走各的机制”很干净，但真实项目里这些维度常会纠缠（比如护栏需要知道身份）。当关注点不得不交叉时，你会怎么保持设计不滑向一团乱麻？
